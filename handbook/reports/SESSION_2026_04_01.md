  - **Infrastructure:** `patient_mapper.dart` — toJson/fromJson
  - **Flutter Intent:** `intervention_intents.dart` — `String? violationTypeId`
  - **Flutter Mapper:** `intervention_mapper.dart` — conversão `String? → LookupId?`

### 2. Migração Isar → Drift

Substituição completa do Isar (abandonado, requer CocoaPods) por Drift (SQLite, ativo, usa build hooks nativas).

#### Camadas migradas

| Pacote | Antes | Depois |
|--------|-------|--------|
| `packages/persistence/` | 3 Isar `@collection` schemas + `.g.dart` gerados | 3 Drift `Table` definitions + `.drift.dart` (modular gen) |
| `packages/core/offline/` | `IsarService` + `SyncQueueService` (Isar filter/put) | `DriftDatabaseService` + `SyncQueueService` (SQL select/insert/update + `watchPendingActions()`) |
| `bff/social_care_desktop/storage/` | `LocalSocialCareRepository` (Isar writeTxn/filter) | Drift select/insert/update/delete/batch |
| `bff/social_care_desktop/sync/` | `SyncEngine` com `Timer.periodic(1 min)` polling | `SyncEngine` com `watchPendingActions().listen()` **reativo** |
| `apps/acdg_system/di/` | `IsarService` nos providers | `DriftDatabaseService` nos providers |
| Dependências | `isar`, `isar_flutter_libs`, `isar_generator` | `drift`, `drift_flutter`, `drift_dev`, `sqlite3_flutter_libs` |

#### Detalhes técnicos
- **Build config:** `build.yaml` com `drift_dev:modular` — builds incrementais ~3s
- **Drift tables:** `CachedPatients`, `SyncActions`, `CachedLookups` com tipos imutáveis
- **Lookup table rename:** `tableName` → `lookupName` (conflito com `Table.tableName` do Drift)
- **Persistence adicionado ao workspace** root (`pubspec.yaml`)

### 3. Polling removido do SyncEngine

- **Antes:** `Timer.periodic(Duration(minutes: 1))` consultava a fila a cada minuto
- **Depois:** `_queueService.watchPendingActions().listen()` — Drift emite automaticamente quando `sync_actions` muda
- **Benefício:** Processamento instantâneo quando action entra na fila, sem queries desnecessárias quando vazia

## Issues encontradas para atacar amanhã

### P0 — Bugs reais

#### 1. `watchPendingActions()` — bug temporal no stream
- **Arquivo:** `packages/core/lib/src/offline/sync_queue_service.dart:57`
- **Problema:** `DateTime.now()` é capturado no momento do `.watch()` e nunca atualiza. Actions com `nextRetryAt` no futuro ficam travadas no stream até que outra mudança na tabela force re-emissão.
- **Solução:** Recriar o watch periodicamente ou usar custom stream que re-avalia `now` a cada emissão do Drift.

#### 2. SyncDetailPanel — violação de camada + naming
- **Arquivo:** `apps/acdg_system/lib/ui/organisms/sync_detail_panel.dart:100-107`
- **Problema 1:** UI chama `isar.db.delete(isar.db.syncActions).go()` diretamente no banco — violação de camada. Deveria delegar para `SyncQueueService.clearAll()` ou similar.
- **Problema 2:** Variável ainda nomeada `isar` quando agora é `DriftDatabaseService`.
- **Solução:** Criar método `clearAll()` no `SyncQueueService`, renomear variável.

### P1 — Padrão `valueOrNull!` no PatientMapper (60+ instâncias)

- **Arquivo:** `bff/shared/lib/src/infrastructure/patient_mapper.dart`
- **Problema:** 60+ usos de `valueOrNull!` nos métodos `fromJson`. Se qualquer `create()` retornar `Failure`, crash em runtime com null force-unwrap.
- **Diretriz violada:** "Evite valueOrNull, sempre prefira exhaustive if com Result usando Success ou Failure"
- **Escopo:** Todos os métodos `*FromJson()` — `personalDataFromJson`, `civilDocumentsFromJson`, `addressFromJson`, `familyMemberFromJson`, `diagnosisFromJson`, `socialIdentityFromJson`, `housingConditionFromJson`, `socioEconomicFromJson`, `socialBenefitFromJson`, `workAndIncomeFromJson`, `educationalStatusFromJson`, `healthStatusFromJson`, `communitySupportFromJson`, `socialHealthSummaryFromJson`, `appointmentFromJson`, `intakeInfoFromJson`, `placementHistoryFromJson`, `violationReportFromJson`, `referralFromJson`
- **Solução:** Refatorar cada método para retornar `Result<T>` com exhaustive pattern matching. Propagar failures ao invés de crashar.

### P2 — Silent failures nos detail mappers

- **Arquivos:**
  - `packages/social_care/lib/src/ui/home/mappers/work_and_income_detail_mapper.dart`
  - `packages/social_care/lib/src/ui/home/mappers/socioeconomic_detail_mapper.dart`
- **Problema:** `.map(...).whereType<Success>()` descarta silenciosamente items que falharam na validação. Usuário recebe dados incompletos sem notificação de erro.
- **Solução:** Coletar failures e retornar `Failure` se algum item falhar, ou ao menos logar um warning.

### P3 — Outras melhorias identificadas

| Item | Arquivo | Problema |
|------|---------|----------|
| `dispose()` sem await | `dependency_manager.dart:91` | `_dbService.close()` chamado sem `await` — pode perder writes pendentes |
| `_mutatePatient` catch genérico | `local_social_care_repository.dart:65` | `catch (e)` perde contexto do erro — deveria wrapping em `AppError` |
| `_dispatchAction` default case | `sync_engine.dart:330` | Retorna `Failure(String)` ao invés de `Failure(AppError)` |
| `listPatients` valueOrNull! | `offline_first_repository.dart:86` | `localResult.valueOrNull!.isNotEmpty` — deveria usar pattern matching |
| Requests duplicadas | `TODO_PENDENCIAS.md` | Bug preexistente — debounce no `processQueue()` ainda pendente |
| `window_to_front` sem SPM | Plugin externo | Substituir por `MethodChannel` nativo (~10 linhas) para eliminar CocoaPods 100% |
