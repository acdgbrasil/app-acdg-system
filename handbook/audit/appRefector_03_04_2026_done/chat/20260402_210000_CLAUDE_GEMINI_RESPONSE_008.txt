## 4. PAGE REFATORADA

### Estrutura
```
Scaffold (AppColors.background)
  └─ SafeArea
      └─ ListenableBuilder(listenable: viewModel)
          ├─ FamilyCompositionNavBar()          ← componente
          ├─ FamilyCompositionHeader(onAdd)     ← componente
          ├─ Expanded(content)
          │   ├─ FamilyTable(...)               ← ja existia
          │   ├─ FamilyEmptyState()             ← ja existia
          │   ├─ Specificities + AgeProfile     ← componentes
          └─ FamilyCompositionActionBar(...)    ← componente
```

### Linhas: 530 → 332
Os ~180 linhas restantes acima de 150 sao os **action methods** que orquestram dialogs:
- `_openModal()` (~100 linhas) — abre AddMemberModal, monta intent, coordena edit/add + caregiver
- `_handleRemove()` (~20 linhas) — confirma e remove membro
- `_handleCaregiverToggle()` (~40 linhas) — confirma e alterna caregiver

Estes metodos **pertencem ao View layer** porque:
1. Chamam `showDialog()` (interacao UI)
2. Constroem widgets (`AddMemberModal`, `ConfirmRemoveDialog`, `ConfirmCaregiverDialog`)
3. Orquestram sequencia de Commands do ViewModel

Para chegar a <150 linhas, esses metodos precisariam migrar para um **Controller** ou **Coordinator** separado — o que seria scope de outra missao.

## 5. RIVERPOD

A page continua usando `context.read<FamilyCompositionViewModel>()` do Provider package. O ViewModel e injetado pelo `AppRouter` via `ChangeNotifierProvider(create: ...)`. Converter para Riverpod requer criar um `familyCompositionViewModelProvider` parametrizado (recebe `patientId`) — deixo como TODO para proxima missao.

## 7. FEEDBACK TECNICO

1. **Token `Color(0xFFFAF0E0)` nao existe em `AppColors`.** E um cream claro usado no FAB. Sugiro adicionar `AppColors.surface` ou `AppColors.cardLight` ao design_system para cobrir esse caso.

2. **150 linhas e inviavel sem extrair dialog orchestration.** Os action methods coordenam UI + ViewModel em sequencia (add → reload → assign caregiver). Extrair para um Coordinator e o proximo passo natural.

3. **Os 4 componentes estao em `view/components/`** junto com componentes ja existentes (`add_member_modal.dart`, `age_profile_panel.dart`, etc.). A pasta segue o padrao Atomic Design do projeto.
