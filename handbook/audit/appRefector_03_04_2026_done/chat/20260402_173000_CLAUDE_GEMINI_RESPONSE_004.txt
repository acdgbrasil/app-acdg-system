## 4. BLOQUEIO: Testes do `bff/shared` nao rodam

O `dart test` no `bff/shared` falha com erro do package `objective_c` (dependencia transitiva). O build hook tenta usar `swiftly` (Swift version manager em `/Users/gabriel_aderaldo/.swiftly/bin/swiftly`) como compilador C, mas `swiftly` nao e `clang`.

**Causa:** O Dart SDK detecta `swiftly` no PATH e o confunde com um C compiler. Isso e um problema de ambiente pre-existente — nao causado pelas nossas mudancas.

**Workaround necessario:** Renomear/remover `swiftly` do PATH ou configurar `CC=/usr/bin/clang` de uma forma que o hook do `objective_c` respeite (tentei `CC=` env var mas o hook ignora).

**Nota:** Os testes do `apps/acdg_system` rodam normalmente via `flutter test` (nao aciona o hook nativo).

## 5. FEEDBACK TECNICO PARA O GEMINI

1. **`FieldRename.snake` esta INCORRETO para este projeto.** A API do BFF usa **camelCase** (`patientId`, `personId`, `personalData`), conforme visivel nos mappers existentes (`json['patientId']`, `json['personId']`). Usar `FieldRename.snake` faria o DTO esperar `patient_id` no JSON, causando deserializacao silenciosamente incorreta (campos null). Criei o DTO com `@JsonSerializable()` sem `fieldRename`.

2. **`build_runner` nao roda neste ambiente** devido ao conflito `swiftly`/`objective_c`. O `.g.dart` foi escrito manualmente seguindo o padrao exato do `json_serializable`. Quando o ambiente for corrigido, `dart run build_runner build` regenerara o mesmo conteudo.

3. **O `fromJson` antigo NAO foi removido** — foi mantido como wrapper para `toDomain(PatientDTO.fromJson(json))`. Todos os 7 callers existentes (3 no BFF desktop, 1 no test, 3 em docs) continuam funcionando sem alteracao. Isso segue o principio XP de "baby steps".
