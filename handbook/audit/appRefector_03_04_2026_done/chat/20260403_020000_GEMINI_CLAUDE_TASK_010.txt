## 1. OBJETIVO (ATACANDO O PONTO 01 DA AUDITORIA)
O Supervisor ordenou e o TDD foi armado: **"TDD em primeiro lugar, nada de legado"**.
Nosso alvo é o uso do `ChangeNotifierProvider` importado de `flutter_riverpod/legacy.dart`. O Riverpod 3.x desencoraja esse provider, e nós não aceitamos "código legado" em um projeto novo.

## 2. A TAREFA (O QUE VOCÊ DEVE FAZER)
Você deve varrer todo o projeto (`packages/social_care` e `apps/acdg_system`) e remover QUALQUER ocorrência de `legacy.dart` e `ChangeNotifierProvider`.

### O Padrão Ouro (Gold Standard) para ViewModels:
Como nossos ViewModels (que estendem `BaseViewModel`) usam `ChangeNotifier` para se comunicar via `Command`, a migração correta para Riverpod 3.x é usar um `Provider` puro e gerenciar o ciclo de vida manualmente.

**ANTES (O que está errado e gera erro no teste):**
```dart
import 'package:flutter_riverpod/legacy.dart'; // PROIBIDO!

final myViewModelProvider = ChangeNotifierProvider.autoDispose<MyViewModel>((ref) {
  return MyViewModel();
});
```

**DEPOIS (O que você DEVE implementar):**
```dart
import 'package:flutter_riverpod/flutter_riverpod.dart';

final myViewModelProvider = Provider.autoDispose<MyViewModel>((ref) {
  final viewModel = MyViewModel(
     // dependencias...
  );
  
  // O Riverpod 3.x gerencia o ciclo de vida e chama o dispose do ChangeNotifier
  ref.onDispose(() {
    viewModel.dispose();
  });
  
  return viewModel;
});
```

### O Impacto na UI:
Como a UI já usa `ConsumerWidget` ou `ConsumerStatefulWidget`, você apenas fará `final vm = ref.watch(myViewModelProvider)`. Como o provedor agora é um `Provider` comum, o `ref.watch` **NÃO** reconstruirá a tela quando `notifyListeners` for chamado (isso é o comportamento correto e desejado). 
A reatividade deve acontecer de forma cirúrgica na tela utilizando `ListenableBuilder(listenable: vm, builder: ...)` em torno dos componentes que precisam atualizar.
Você já aplicou isso na `HomePage`! O modelo mental é o mesmo para as outras telas.
