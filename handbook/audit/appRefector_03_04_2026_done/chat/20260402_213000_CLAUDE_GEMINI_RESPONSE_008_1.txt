## 2. RIVERPOD PROVIDER

**Arquivo:** `apps/acdg_system/lib/logic/di/social_care_providers.dart`

```dart
final familyCompositionViewModelProvider = ChangeNotifierProvider.autoDispose
    .family<FamilyCompositionViewModel, String>((ref, patientId) {
  return FamilyCompositionViewModel(
    patientId: patientId,
    getPatientUseCase: ref.watch(getPatientUseCaseProvider),
    addFamilyMemberUseCase: ref.watch(addFamilyMemberUseCaseProvider),
    removeFamilyMemberUseCase: ref.watch(removeFamilyMemberUseCaseProvider),
    updatePrimaryCaregiverUseCase: ref.watch(updatePrimaryCaregiverUseCaseProvider),
    lookupRepository: ref.watch(lookupRepositoryProvider),
  );
});
```

- `autoDispose` — limpa quando a rota sai do stack
- `.family<..., String>` — parametrizado por `patientId`
- Colocado no shell (`acdg_system`) onde os use case providers ja existem
- Usa `ChangeNotifierProvider` do `legacy.dart` (necessario enquanto VM herda ChangeNotifier)

## 5. NOTA SOBRE PROVIDER VS RIVERPOD NA PAGE

A page ainda usa `context.read<FamilyCompositionViewModel>()` do Provider package. O `familyCompositionViewModelProvider` esta pronto no shell, mas a page e injetada pelo `AppRouter` via `ChangeNotifierProvider(create: ...)`. Para migrar completamente, o route builder do `AppRouter` precisa usar `Consumer` + `familyCompositionViewModelProvider` — escopo de proxima missao de roteamento.

## 7. FEEDBACK TECNICO

1. **189 > 150 linhas** — os ~40 linhas extras sao os 3 dialog trigger methods (`_showAddModal`, `_showRemoveDialog`, `_showCaregiverDialog`). Eles sao "View pura" (chamam `showDialog`), nao logica de negocio. Para chegar a 150 seria necessario mover os dialogs para widgets que se auto-exibem via state do ViewModel — overengineering neste momento.

2. **`ChangeNotifierProvider` do `legacy.dart`** e necessario no `familyCompositionViewModelProvider` porque o VM herda de `BaseViewModel` → `ChangeNotifier`. Quando o VM migrar para `Notifier`, o `legacy.dart` desaparece.

3. **Teste TDD do Gemini nao testa os novos metodos** (`handleModalSave`, `handleRemove`, etc.). Sugiro um teste adicional para cobrir esses fluxos.
