## 8. CORRECOES ARQUITETURAIS (009.2)

Apos rejeicao 009.2, tres violacoes comportamentais corrigidas:

### 8.1 FamilyMemberModal — Entry externalizado
- **ANTES:** `_entry = FamilyMemberEntry()` instanciado no `initState()` do Widget
- **DEPOIS:** `FamilyMemberEntry entry` passado via construtor — o caller (`StepFamilyCompositionContent`) cria e configura o entry antes de abrir o modal
- O modal agora e 100% burro: le de `widget.entry`, nao cria estado

### 8.2 FamilyMemberModal — ScaffoldMessenger removido
- **ANTES:** `ScaffoldMessenger.of(context).showSnackBar(...)` direto no modal para conflito de caregiver
- **DEPOIS:** `VoidCallback? onCaregiverConflict` callback — o caller decide como notificar. O modal apenas chama `widget.onCaregiverConflict?.call()`

### 8.3 HomePage — setState eliminado
- **ANTES:** `String _activeTab` no State + `setState(() => _activeTab = tab)` + `onChanged: () => setState(() {})`
- **DEPOIS:** `activeTab` e `onSearchChanged()` movidos para `HomeViewModel`. A page usa `viewModel.setActiveTab(tab)` e `viewModel.onSearchChanged`. Zero `setState` na page.

## 9. CORRECOES SRP (009.3)

Re-exports adicionados no `family_composition_form_state.dart` para backward compat:
```dart
export '../../../../models/family_member_snapshot.dart';
export 'family_member_entry.dart';
```

### 9.2 Inversao de Controle no Modal

**ANTES:** View criava e preenchia o `FamilyMemberEntry`:
```dart
final entry = FamilyMemberEntry();
entry.name.text = existing.name; // View preenchendo dados!
```

**DEPOIS:** FormState cria o entry pre-preenchido:
```dart
final entry = existing != null
    ? formState.createEntryFromSnapshot(existing)
    : formState.createEntry();
```

O modal recebe o entry **pronto** — nao preenche nada.

Factory method `FamilyMemberEntry.fromSnapshot(snapshot)` encapsula toda a logica de preenchimento.
