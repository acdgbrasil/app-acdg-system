## 7. FEEDBACK TECNICO — DIVERGENCIAS DA PROPOSTA

### 7.1 `fetchPatient({required String byID})` — NAO IMPLEMENTADO
**Motivo:** Regressao de type safety. `PatientId` e um Value Object validado (UUID format + factory `create` com Result). Usar `String` permite UUIDs malformados passarem sem validacao. Mantive `fetchPatient(PatientId id)`.

Se o Gemini insistir no `String`, sugiro criar um overload:
```dart
Future<Result<PatientRemote>> fetchPatientByRawId(String rawId);
```
Que valida internamente via `PatientId.create(rawId)`.

### 7.2 `byID` vs `byId`
Dart style guide: acronimos de 2 letras em camelCase usam lowercase (`byId`). `byID` e convencao Swift. Mantive sem named parameter por ora — o metodo `fetchPatient(PatientId id)` e autodocumentado pelo tipo.

### 7.3 `PatientRepository` (app layer) NAO foi renomeado
A interface `PatientRepository` em `packages/social_care/` e uma camada ACIMA do `SocialCareContract`. Seus metodos (`listPatients`, `getPatient`) sao a API do app layer, nao do BFF. Renomea-los seria escopo de outra missao e afetaria UseCases, ViewModels, e toda a UI.

### 7.4 Testes existentes precisam de atualizacao separada
~30 arquivos de teste referenciam nomes antigos (`PatientMapper`, `getPatient`, `listPatients`). Estes devem ser atualizados numa missao de testes dedicada para evitar escopo excessivo.
